IE (0xFFFF) is which sources the game has opted into; IF (0xFF0F) is which sources currently have a request pending.

How the PPU spends its time drawing one frame. A frame is 154 scanlines × 456 T-cycles each = 70,224 T-cycles (~16.74ms, ~59.73Hz). Each of the 144 visible lines steps through 3 modes before 10 V-Blank lines close out the frame.

Mode 2: OAM Search (80T) Mode 3: Pixel Transfer (172T) Mode 0: H-Blank (204T) Mode 1: V-Blank (whole line)

Per line: Mode 2 80T (~19.1µs) → Mode 3 172T (~41.0µs) → Mode 0 204T (~48.6µs)  =  456T (~108.7µs) total.
V-Blank: 10 lines × 456T = 4,560T (~1.09ms) — VRAM/OAM are free to touch from the CPU for the whole stretch, which is why games do most of their per-frame logic and OAM DMA here.
This emulator uses fixed-length modes for simplicity; real hardware's Mode 3 can stretch a little longer depending on sprite count, SCX fine-scroll, and the window, which is why Mode 0's length isn't perfectly constant on a real Game Boy.

Raw contents of tile data in VRAM (0x8000–0x97FF), decoded as 384 8×8 tiles in the order in VRAM. Always shown palette-agnostic/grayscale, on both GB and GBC, since a tile has no fixed color until it's placed in a map or OAM entry.

In VRAM. The full 32×32 tile (256×256 pixel) map. On GB, rendered with the current BGP register. Not limited to VRAM: point it at ROM or WRAM to preview graphics data before it's ever copied to VRAM. On GBC, a VRAM address reads whichever of the two banks is currently switched in via VBK.

Frame Activity

What the emulated Game Boy actually did, frame by frame — not real-world JS/host timing, but counts of hardware events inside the simulation itself. Every frame takes exactly 70,224 T-cycles (154 scanlines × 456 T-cycles ≈ 16.74ms @ 4.194304MHz, ~59.73Hz) no matter what happens during it, so the numbers here describe how that fixed budget was spent — not how long the frame took, since it always takes the same amount of time. Click a frame on the left to inspect where in that budget its events happened, on the right.

Bar height = instructions executed that frame (relative to the tallest bar in view, not an absolute count). Since every frame spends the same 70,224 T-cycles regardless, a shorter bar doesn't mean less happened — it usually means the CPU ran longer instructions that frame (up to 24T each), or spent more of the frame HALTed.

Click a scanline in the "Anatomy of frame" chart above to see its details.

Frame budget: 144 visible lines × 456T = 65,664T (93.5%) + 10 V-Blank lines × 456T = 4,560T (6.5%) = 70,224T total. See the Scanline Timeline tool above for how each visible line's 456T further splits into Mode 2 (OAM Search) / Mode 3 (Pixel Transfer) / Mode 0 (H-Blank).
OAM DMA: one transfer takes 160 M-cycles = 640T (~1.4 scanlines, ~0.9% of a frame); the CPU can only touch HRAM while it runs, which is exactly why games trigger it from inside their V-Blank handler rather than mid-screen.
Interrupt dispatch: ~5 M-cycles = 20T (~0.03% of a frame) of fixed hardware overhead per interrupt taken — pushing PC to the stack and jumping to the handler — before the handler's own code even starts running. Interrupts are serviced in a fixed priority order (V-Blank > STAT > Timer > Serial > Joypad) if more than one is pending at once.
Instructions: GB instructions take 1-6 M-cycles (4-24T) each, so the ~17,556 M-cycles available in a frame fit roughly 3,000-6,000 instructions depending on which ones run and how much of the frame the CPU spends HALTed waiting for the next interrupt.
